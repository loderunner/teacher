# Story 5 — Chapter Chat Persistence (D3.5)

## Context

Stories 1–4 of D3 (see `.claude/plans/03-01-chapter-page.md`,
`.claude/plans/03-02-chapter-chat.md`, `.claude/plans/03-03-chapter-completion.md`,
and `.claude/plans/03-04-syllabus-change.md`) have shipped a fully functional
chapter destination: the journey URL redirects to the active chapter, the
chapter page renders a streaming `ChapterChat` client island, the chat wires
three tools (`updateMemory`, `markChapterComplete`, `proposeSyllabusChange`),
and the model can drive both chapter completion and syllabus restructuring
through user-confirmed server actions. The one thing the chapter chat does
**not** do yet is survive a refresh — every prior story explicitly carved out
chat history as "ephemeral, like the welcome chat."

Story 5 closes that gap. Returning to a chapter page — by navigating back, by
refreshing, or by following the syllabus panel to a done chapter — restores
the full conversation history from the database, so the learner can continue
exactly where they left off. No new tools, no new UI, no new model calls.
Just DB-backed durability for the chapter chat that Stories 2–4 have built.

Specifically:

- Refresh mid-chapter → all prior turns reload from the DB, including the
  assistant's tool-call parts (the inline "Go to next chapter" button from
  Story 3 and the inline proposal card from Story 4 both re-render). Streaming
  continues from the next user message.
- Navigate to another chapter, then come back → that chapter's full history
  is restored.
- Complete a chapter via Story 3, then return to the done chapter via the
  syllabus panel → still sees the full history.

This also lets us **remove the "client passes messages to the server action"
caveat** documented in Story 3: `completeChapterAction` no longer accepts a
`messages` array — it reads them from the DB. The server-side message store
becomes the single source of truth for any future server action that needs
the transcript.

What stays deliberately out of scope:

- **Delta message transport for chapter chat.** Stories 2–4 already send a
  full validated `messages: UIMessage[]` body on each turn; Story 5 keeps
  that wire format unchanged (no `prepareSendMessagesRequest`, no
  `{ message, regenerateFromMessageId }` body). Narrowing the client payload
  and teaching the route to assemble history from DB alone is the separate
  [Delta Message Transport plan](../../docs/plans/delta-message-transport.md)
  — for chapter chat, that plan’s §11 Step 5 ships **after** Story 5 proves
  persistence with the default transport.
- **Welcome (syllabus) chat persistence.** The `phase` enum on the new
  `messages` table includes `'syllabus'` for future use, but Story 5 only
  ever writes `phase='chapter'` rows. The welcome chat stays ephemeral per
  `00-journey-app.md` line 209.
- **Persisting `tool-proposeSyllabusChange` parts at all.** The chapter-chat
  `onFinish` hook strips any `tool-proposeSyllabusChange` parts from the
  assistant message before saving, so a refresh never re-renders an
  unresolved proposal card. Proposals are by design one-shot, in-session
  interactions: the model fires, the user accepts or dismisses, the page
  moves on. The applied effect (a syllabus change) lives in the journey's
  data; the un-applied card has no business reappearing later. This also
  removes the need to persist Story 4's session-local dismissal state.
- **Persistence during streaming.** The assistant message is saved in a
  single write inside `onFinish`. If `onFinish` errors after the stream has
  reached the client, the user sees the assistant's reply but the DB has
  only the user message; a refresh would lose the assistant turn. This is
  the only durability gap and is logged at `error` level. A more robust
  fix (incremental saves) is deferred.

Story 5 is the final story of D3. The next milestone is D5 (the `/journeys`
list page) per `00-journey-app.md`.

## Decisions

- **New `messages` table.** Lands as a single Drizzle migration generated by
  `pnpm drizzle-kit generate`. Columns mirror the `00-journey-app.md` sketch
  with two deliberate deviations documented below. Indexes: a composite
  `(chapterId, createdAt)` index for the primary read pattern (load all
  messages for a chapter, ordered) and a `(journeyId, phase, createdAt)`
  index for any future syllabus-phase reads.

- **Schema deviation #1: `role` enum drops `'tool'`.** The AI SDK
  `UIMessage.role` is `'user' | 'assistant' | 'system'`. The `'tool'` role
  from the `00-journey-app.md` sketch is a leftover from older AI SDK shapes
  — tool calls now live as parts inside an assistant message, not as
  standalone messages. The new `chat_role` enum is
  `('user' | 'assistant' | 'system')`; the deviation is called out in a
  comment on the schema.

- **Schema deviation #2: `messages.id` is the AI SDK `UIMessage.id`,
  not a server-side nanoid.** `useChat` generates an id per message via
  `generateId()` and round-trips it across the wire. Using it as the PK
  enables idempotent saves via `ON CONFLICT (id) DO NOTHING`, which covers
  client retries and double-submits. Assigning a separate nanoid would
  break idempotency and require a side-table mapping. JSDoc on the column
  documents the contract.

- **What goes in `parts`.** The AI SDK `UIMessage['parts']` array verbatim
  — text parts, tool-call parts (including their input + state), reasoning
  parts. Same shape `useChat` produces and consumes. Stored as `jsonb`,
  notNull.

- **Persistence trigger: server-side, in the chat route handler.** The
  Story 2 route gains two new write points around the existing `streamText`:
  1. Before streaming, save the **incoming new user message** (the last
     message in the validated `messages` array). Idempotent on `id`. This
     must succeed; a save failure returns 500 before any model call.
  2. Inside `result.toUIMessageStreamResponse({ onFinish })`, save the
     **assistant response message** (also idempotent on `id`). `onFinish`
     fires after the model has finished and receives the complete
     `responseMessage` with all tool parts attached. The save runs even on
     a stream-consumer disconnect.

  Why server-side and not client-side: the client could fail to post a save
  request after a network drop or refresh during streaming. Server-side
  persistence is the only way to guarantee durability of the assistant's
  completed message. Doing the user-message save in the same request as
  the model call keeps a single ordering source of truth.

- **Wire shape: unchanged — full `messages` array per Stories 2–4.** The
  client keeps using `DefaultChatTransport`’s default body: validated
  `messages: UIMessage[]` plus `locale` (and any other fields the route
  already accepts). The server still builds `modelMessages` from that array
  via `convertToModelMessages` — the same shape `useChat` hydrates from
  `initialMessages` after a reload. Story 5 does **not** add
  `prepareSendMessagesRequest` or switch the chapter route to the delta
  request schema; that work belongs solely to the
  [Delta Message Transport plan](../../docs/plans/delta-message-transport.md).

- **Persistence trigger: reconcile DB tail, then save terminal user turn.**
  After `validateUIMessages`, before `streamText`, the route ensures the
  persisted chapter transcript matches the client-supplied history **for
  purposes of durability**:
  1. When the validated array implies a truncated tail (edited user message
     or regeneration that removed the last assistant turn), call
     `deleteMessagesFrom` so superseded rows are removed — same semantics as
     the companion delta plan §5 (the primitive is shared; only the HTTP
     wire format differs).
  2. When the last validated message is a `user` message, call
     `saveChatMessage` for it (idempotent on `id`). Skip when the array is
     empty (assistant-first turn — nothing new to persist before the model
     runs).

  Idempotency remains `ON CONFLICT (id) DO NOTHING` on `saveChatMessage`.

- **`triggerResponse` / assistant-first turn.** `ChapterChat` calls
  `triggerResponse()` on mount, which calls `sendMessage(undefined)`. The AI
  SDK issues a request whose validated `messages` array is empty (same as
  Stories 2–4). No user row is saved before `streamText`; `onFinish` still
  persists the assistant reply. After Story 5, a mid-chapter refresh reloads
  `initialMessages` from the DB so the client no longer sends an empty array
  on remount — the first request in a resumed session carries the restored
  history instead.

- **Synthetic user message from `applySyllabusChangeAction`.** `ChapterPage`
  injects a synthetic `{ role: 'user', parts: [{ type: 'text', text: '...' }] }`
  message into `useChat` state via `setMessages` after a syllabus proposal is
  applied (Story 4). That message must also be written to the DB or the next
  turn after a refresh would miss the "I applied the suggested syllabus
  change" context. Resolution: `applySyllabusChangeAction` generates the
  message ID server-side (via `nanoid`), saves the synthetic message to the DB
  with `saveChatMessage` (using `getLocale()` for the translated text), and
  returns the `id` to the client. `handleSyllabusApplied` in `ChapterPage`
  receives the `id` and uses it as the `id` field on the injected `setMessages`
  entry so that client state and DB are in sync. If the save fails the action
  returns without the id and the client falls back to `crypto.randomUUID()` —
  the context gap reappears but does not break the UI.

- **Loading history on the chapter page.** The chapter page (server
  component, Story 1) gains a new server-side fetch:
  `listChapterMessages({ userId, journeyId, chapterId })` returning
  `UIMessage[]` ordered by `createdAt`. The result is passed to
  `<ChapterChat>` as a new `initialMessages` prop, which forwards into
  `useChat({ messages: initialMessages, ... })`. The AI SDK v6 `ChatInit`
  type names the prop `messages?: UI_MESSAGE[]`.

- **`useChat` hydration is the same shape `useChat` writes on submit.**
  Because `messages.id` is stored verbatim, the round-trip
  DB → `initialMessages` → `useChat` state → POST body → DB never mints a
  new id for an existing message. Stable ids on reload also prevent the
  `<Message key={msg.id}>` list in `ChatScaffold` from remounting.

- **Two new entity functions, under a fresh `lib/server/messages/` package.**
  The new package mirrors the conventions of `lib/server/chapters/` (created
  in Story 3) and `lib/server/journeys/`. Both functions are pure and
  REST-ready per `AGENTS.md` — no Next.js concerns.
  - `lib/server/messages/list.ts` — `listChapterMessages({ userId, journeyId, chapterId })`.
    Joins through `journeys.userId` so a request for a chapter belonging to
    another user returns `[]` (not an error), matching the rest of the
    entity layer's "scoped UPDATE / scoped SELECT" idiom.
  - `lib/server/messages/save.ts` — `saveChatMessage({ userId, journeyId, chapterId, phase, message })`.
    Idempotent via `.onConflictDoNothing({ target: messages.id })`. The
    ownership gate is a single `select` of the journey by `(id, userId)`
    inside the same call; on miss, the function returns without writing.

- **New entity function `deleteMessagesFrom`.** Required so edits and
  regenerations stay consistent with the DB while the wire format still
  carries a full `messages` array. Lives in `lib/server/messages/delete.ts`.
  Deletes all messages
  in a chapter scope with `created_at >=` the `created_at` of the row whose
  `id` matches `fromMessageId`. Safe when the id does not exist (PostgreSQL
  `>= NULL` is falsy; zero rows deleted). Full spec in the companion Delta
  Message Transport plan §5 (same SQL when chapter delta ships later). Added
  to the `lib/server/messages/` barrel.

- **Phase is hard-coded at all Story 5 call sites.** The route handler and
  every action that saves a message passes `phase: 'chapter'`. The enum
  carries `'syllabus'` for future use only.

- **`completeChapterAction` cleanup.** Drop the `messages` field from
  `CompleteChapterInput`. The action now loads the transcript via
  `listChapterMessages({ userId, journeyId: journey.id, chapterId: chapter.id })`
  server-side and passes the result to `generateChapterSummary`. The
  Story 3 contract for `generateChapterSummary` — `{ style, locale,
  chapter, messages }` — is preserved; only the *caller* changes. The
  "client passes messages" security caveat documented in Story 3's
  Decisions is removed in the same diff. `chapter-chat.tsx`'s
  `handleComplete` callback no longer forwards `messages` in the action
  call.

- **`applySyllabusChangeAction` (Story 4) is unchanged.** It never read
  messages.

- **`tool-proposeSyllabusChange` parts are stripped before save.** The
  chapter-chat `onFinish` filters them out of `responseMessage.parts`
  before calling `saveChatMessage`. A reloaded chapter never re-renders
  an unresolved proposal card. Story 4's session-local
  `dismissedProposals: Set<string>` state is still used for in-session
  dismissal (clicking Dismiss removes the card *during* the conversation)
  but it has no role across reloads, because the underlying tool part
  never lands in the DB. If the filter leaves the assistant message with
  zero parts (the model fired only `proposeSyllabusChange` and emitted no
  text, which is the system-prompt-mandated shape), skip the insert
  entirely — there is nothing to persist.

- **Reloaded `tool-markChapterComplete` button is idempotent on the
  server.** A refreshed page that previously fired `markChapterComplete`
  still renders the inline "Go to next chapter" button. Clicking it now
  hits `completeChapterAction` against a chapter whose `status` is already
  `'done'`; Story 3's entity function `completeChapter` short-circuits to
  the idempotent branch and returns the existing `nextIdx`. The UX still
  works: the user is sent forward without re-generating a summary.

- **The welcome-chat stays ephemeral.** `app/api/syllabus/chat/route.ts`
  and `app/[locale]/_components/welcome-chat.tsx` are explicitly **not**
  modified by Story 5. The `phase='syllabus'` enum branch exists only for
  future expansion.

- **Single-statement writes; no transaction needed.** Both saves are
  single inserts. The `db` (HTTP) client is sufficient; `dbTx` (WS) is
  not required.

- **Save failures.** The user-message save runs before `streamText`; on
  failure, the route returns 500 and no model call happens. The
  assistant-message save inside `onFinish` runs *after* the stream has
  reached the client, so any error there only affects durability — the
  user sees the response, but a refresh loses it. Errors at both points
  are logged at `error` level. No retries, no incremental persistence in
  Story 5.

- **No new tests.** `messages/parity.test.ts` (i18n key parity) and
  `lib/url.test.ts` (Story 1) continue to pass unchanged. Story 5 adds no
  new i18n keys (the UI surface is identical) and no new pure helpers worth
  unit-testing. Verification is manual end-to-end.

### 1. `lib/server/db/schema.ts` — add the `messages` table + `chat_phase` and `chat_role` enums

Append after the `chapters` table. New enums use `pgEnum` like
`chapterStatusEnum`. The migration generated by `pnpm drizzle-kit generate`
creates both Postgres enums (`chat_phase`, `chat_role`), the `messages`
table, the foreign keys with cascade-on-delete, and the two indexes.

```ts
/** Postgres enum for the chat phase a message belongs to. */
export const chatPhaseEnum = pgEnum('chat_phase', ['syllabus', 'chapter']);

/**
 * Postgres enum for the role of a chat message.
 *
 * Note: deviates from the original `00-journey-app.md` sketch which listed
 * a `'tool'` value. Modern AI SDK `UIMessage` has no `'tool'` role — tool
 * calls live as parts inside an assistant message.
 */
export const chatRoleEnum = pgEnum('chat_role', ['user', 'assistant', 'system']);

/**
 * Database table for persisted chat messages.
 *
 * The primary key is the AI SDK `UIMessage.id` from the client — this makes
 * save calls idempotent under `ON CONFLICT (id) DO NOTHING` and keeps the
 * id stable across reload → `useChat` → submit round-trips.
 */
export const messages = pgTable(
  'messages',
  {
    id: text('id').primaryKey(),
    journeyId: text('journey_id')
      .notNull()
      .references(() => journeys.id, { onDelete: 'cascade' }),
    chapterId: text('chapter_id').references(() => chapters.id, {
      onDelete: 'cascade',
    }),
    phase: chatPhaseEnum('phase').notNull(),
    role: chatRoleEnum('role').notNull(),
    parts: jsonb('parts').notNull(),
    createdAt: timestamp('created_at').notNull().defaultNow(),
  },
  (t) => [
    index('messages_chapter_idx').on(t.chapterId, t.createdAt),
    index('messages_journey_phase_idx').on(t.journeyId, t.phase, t.createdAt),
  ],
);
```

JSDoc on `chapterId` notes it is nullable to leave room for
`phase='syllabus'` rows in the future, but Story 5 always writes a non-null
`chapterId`. Run `pnpm drizzle-kit generate` after saving the schema —
never hand-edit `_journal.json` or the snapshot files.

### 2. `lib/server/messages/list.ts` — new entity function

Mirrors `lib/server/chapters/complete.ts` for the ownership-gate-via-parent
idiom (Story 3) but uses the HTTP `db` client (single SELECT).

```ts
import type { UIMessage } from 'ai';
import { and, asc, eq } from 'drizzle-orm';

import { db } from '@/lib/server/db';
import { chapters, journeys, messages } from '@/lib/server/db/schema';

/** Parameters for loading the chapter-scoped chat history. */
export type ListChapterMessagesInput = {
  /** Clerk user ID — scopes the read to the owner of the journey. */
  userId: string;
  /** Journey the chapter belongs to. */
  journeyId: string;
  /** Chapter whose history is requested. */
  chapterId: string;
};

/**
 * Loads the persisted chat history for one chapter, ordered by `createdAt`.
 *
 * The read is scoped to the owner via a join through `journeys.userId` so
 * a request for a chapter belonging to another user resolves to an empty
 * array rather than leaking rows. Returns shape matches what `useChat`
 * expects as `initialMessages`.
 *
 * @param input - Owner ID, journey ID, and chapter ID.
 * @returns Ordered chat history as `UIMessage[]`.
 */
export async function listChapterMessages({
  userId,
  journeyId,
  chapterId,
}: ListChapterMessagesInput): Promise<UIMessage[]> {
  const rows = await db
    .select({
      id: messages.id,
      role: messages.role,
      parts: messages.parts,
    })
    .from(messages)
    .innerJoin(journeys, eq(messages.journeyId, journeys.id))
    .innerJoin(chapters, eq(messages.chapterId, chapters.id))
    .where(
      and(
        eq(journeys.id, journeyId),
        eq(journeys.userId, userId),
        eq(chapters.id, chapterId),
        eq(messages.chapterId, chapterId),
        eq(messages.phase, 'chapter'),
      ),
    )
    .orderBy(asc(messages.createdAt));

  const items: UIMessage[] = rows.map((row) => ({
    id: row.id,
    role: row.role,
    parts: row.parts as UIMessage['parts'],
  }));
  return items;
}
```

The `parts as UIMessage['parts']` is the single tolerated boundary cast —
`jsonb` is typed as `unknown` by Drizzle, and AI SDK doesn't ship a runtime
validator suitable for "trust your own DB rows." If a project-wide
predicate appears later, replace the cast.

### 3. `lib/server/messages/save.ts` — new entity function

Mirrors `lib/server/journeys/setStyle.ts` for the scoped-by-owner shape but
adds the chapter-existence gate before insert. Uses idempotent insert.

```ts
import type { UIMessage } from 'ai';
import { and, eq } from 'drizzle-orm';

import { db } from '@/lib/server/db';
import { chatPhaseEnum, journeys, messages } from '@/lib/server/db/schema';

/** Parameters for persisting a single chat message. */
export type SaveChatMessageInput = {
  /** Clerk user ID — scopes the ownership check to the journey owner. */
  userId: string;
  /** Journey the message belongs to. */
  journeyId: string;
  /** Chapter the message belongs to (Story 5 always sets this). */
  chapterId: string;
  /** Chat phase (Story 5 always passes `'chapter'`). */
  phase: (typeof chatPhaseEnum.enumValues)[number];
  /** AI SDK UIMessage to persist. The `id` is used as the row PK. */
  message: UIMessage;
};

/**
 * Persists one chat message. Idempotent on `messages.id` so a retry or
 * double-submit with the same id is a no-op.
 *
 * The owner gate runs first; if the journey is missing or owned by a
 * different user the call returns without writing.
 *
 * @param input - Owner ID, journey ID, chapter ID, phase, and message.
 */
export async function saveChatMessage({
  userId,
  journeyId,
  chapterId,
  phase,
  message,
}: SaveChatMessageInput): Promise<void> {
  const owned = await db
    .select({ id: journeys.id })
    .from(journeys)
    .where(and(eq(journeys.id, journeyId), eq(journeys.userId, userId)));
  if (owned.length === 0) {
    return;
  }

  await db
    .insert(messages)
    .values({
      id: message.id,
      journeyId,
      chapterId,
      phase,
      role: message.role,
      parts: message.parts,
    })
    .onConflictDoNothing({ target: messages.id });
}
```

The owner gate uses the same shape as `completeChapter`'s gate in Story 3.
The function intentionally returns `void` even on no-op — callers should
not branch on success vs. silent skip.

### 4. `app/api/journeys/[id]/chapters/[n]/chat/route.ts` — persist around existing full-body flow

Keep `RequestBody` as `{ messages: UIMessage[]; locale: Locale }` and retain
`validateUIMessages`. Add imports for persistence helpers:

```ts
import { deleteMessagesFrom } from '@/lib/server/messages/delete';
import { saveChatMessage } from '@/lib/server/messages/save';
```

After validation and before `convertToModelMessages(messages)`:

1. **Reconcile the DB tail** so persisted rows match the client-supplied
   transcript for edits/regenerations (same outcomes as the delta-plan §4
   server algorithm, but inferred from the full `messages` array rather than
   from `message` / `regenerateFromMessageId` fields). Concretely: when the
   last validated message is `user`, call `deleteMessagesFrom` from that
   message’s `id` (truncates nothing for a brand-new id; truncates the fork
   for an edit). When the client omitted a trailing assistant message that
   still exists in the DB (regeneration), call `deleteMessagesFrom` from the
   regenerated assistant’s `id` before saving anything else. Order these
   branches so a save failure never leaves the DB ahead of the model call in
   an inconsistent way; a failed `saveChatMessage` for the terminal user turn
   returns 500 before `streamText`.

2. **Save the terminal `user` message** when present (idempotent on `id`).
   Skip when `messages.length === 0` (assistant-first mount on a fresh
   chapter).

Build `modelMessages` from the **validated client `messages`** via
`convertToModelMessages` — unchanged from Stories 2–4. The `'Begin.'` cue stays
tied to `history.length === 0` after conversion (empty client array on first
visit).

Pass an `onFinish` handler to `result.toUIMessageStreamResponse` — same as
today, extended with the `tool-proposeSyllabusChange` strip and
`saveChatMessage` for the assistant:

```ts
return result.toUIMessageStreamResponse({
  onFinish: async ({ responseMessage }) => {
    const persistedParts = responseMessage.parts.filter(
      (p) => p.type !== 'tool-proposeSyllabusChange',
    );
    if (persistedParts.length === 0) {
      return;
    }
    try {
      await saveChatMessage({
        userId,
        journeyId: journey.id,
        chapterId: chapter.id,
        phase: 'chapter',
        message: { ...responseMessage, parts: persistedParts },
      });
    } catch (error) {
      console.error('Failed to persist assistant message', error);
    }
  },
});
```

The `onFinish` callback still fires after the stream is flushed; the save
failure path is unchanged from the narrative above.
